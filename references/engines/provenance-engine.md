# Provenance Engine

The anti-hallucination core of CertiCoach. Every command that asserts a
fact about an exam, a resource, or a question must route that fact through
this engine before it is shown to the user or written to state. No other
engine or command is permitted to invent a shortcut around these rules.

Produces the rules `label_fact(fact, tier, source, date)`,
`may_persist(fact)`, and `version_alert(old_facts, new_facts)`. Consumed by
`discover`, `resources`, and the session protocol in `SKILL.md` — any
command that touches `exam-facts.md` or `resources.md` calls into this
engine rather than re-deriving its own labeling or persistence logic.

---

## Content tiers

Every factual or practice-content line shown to the user or written to
`.certicoach/` state carries exactly one of three tags, reproduced here
verbatim:

- `[OFFICIAL — source + date]` — the fact came directly from the
  certification provider's own page (exam guide, syllabus PDF, pricing
  page, exam-code registry, official blog/changelog). Use this tag only
  when both a source URL and a retrieval date are known. This is the only
  tier that may populate `exam-facts.md`'s core sections (Identity,
  Format, Passing policy, Cost & logistics, Syllabus/domains, Recent
  changes).

- `[COMMUNITY — source]` — the fact is a clarifying detail, tip, or
  experience report from a non-official source (forum thread, blog post,
  study group, third-party review). It supplements official facts but
  never overrides them, and it is never used to state something the
  official source does not confirm (e.g. passing score, question count).
  Community facts live in `exam-facts.md`'s "Community notes" section and
  in `resources.md` entries tagged as community material, never mixed
  into the official sections above.

- `[GENERATED PRACTICE]` — the content was generated by CertiCoach itself
  (a practice question, a scenario, an explanation drafted to teach a
  concept). It is not a claim about the real exam and must never be
  presented, logged, or persisted as if it were. `question-attempts.md`
  and `mistake-ledger.md` rows for generated content always carry this
  tag so the user can never mistake a generated question for a leaked or
  official one.

`label_fact(fact, tier, source, date)` is the operation every producing
command calls: given a candidate fact, it selects one of the three tags
above and attaches the source (and date, when the tier requires one)
before the fact is allowed to leave this engine.

## Persistence rule

`may_persist(fact)` returns true only when the fact carries a source URL
**and** a retrieval date. This is a hard gate, not a style preference:

- A fact with a source URL and a retrieval date may be written to
  `exam-facts.md` (or any other state file) tagged
  `[OFFICIAL — <url> — retrieved YYYY-MM-DD]`.
- A fact with a non-official source but no date requirement may be written
  tagged `[COMMUNITY — <source>]` — community tier does not require a
  date, but it must still name a concrete source, not "I recall" or "it's
  commonly known."
- A fact with **no source at all** — something recalled from training data,
  inferred, or guessed — is never written to `exam-facts.md`. Do not
  persist it, do not present it as settled fact, and do not silently drop
  the request either. Instead, tell the user plainly that this detail
  could not be verified from a current source, and ask them to paste the
  relevant section of the official exam page so it can be labeled and
  persisted correctly.

This rule applies uniformly regardless of how confident the underlying
model is about the fact. Confidence is not a substitute for a citation.

## Version-awareness alerts

`version_alert(old_facts, new_facts)` compares the previously persisted
`exam-facts.md` (or resource set) against newly discovered official
information and fires an alert to the user whenever one of these four
triggers is detected:

1. **Syllabus change** — the domain list, domain weightings, or objectives
   under "Syllabus / domains" differ between `old_facts` and `new_facts`.
2. **Exam retiring** — the `Status` field moves from `active` to
   `retiring on YYYY-MM-DD` or `retired`, or a retirement date newly
   appears.
3. **New exam code** — the provider has replaced the exam code tracked in
   `old_facts` with a different code (a successor exam), even if the exam
   name is unchanged.
4. **Previously-selected materials outdated** — an entry in
   `resources.md` was version-matched to the prior `Exam version matched`
   value, and that version no longer corresponds to the current syllabus
   revision in `new_facts`.

On any of these triggers, the alert is surfaced to the user at the next
session boundary (never silently absorbed into the plan) with a plain-
language explanation of what changed and what it means for their current
plan and resource set — this is what lets `plan` and `resources`
recompute instead of continuing to teach or curate against a stale
target.

## No-web degradation

When web access is unavailable — no search tool, no fetch, or all fetch
attempts fail — this engine refuses to assert exam facts from memory
alone. Concretely:

- `discover`, `resources`, and any other command that would normally call
  this engine to label a freshly-fetched fact instead stops before
  writing anything to `exam-facts.md` or `resources.md`.
- The user is told explicitly that web access is unavailable and that no
  facts can be verified right now, rather than being served an answer
  that looks sourced but isn't.
- The user is asked to paste the text of the official exam page (or the
  specific section needed) so the engine can label the pasted content
  `[OFFICIAL — <url the user provides> — retrieved YYYY-MM-DD]` and
  proceed with persistence under the normal rule above.
- If the user has no official page to paste and none can be fetched, the
  engine does not fall back to `[COMMUNITY]` or unlabeled content to fill
  the gap — it leaves the relevant field absent and moves on, rather than
  guessing to appear complete. Incomplete-but-honest state is preferred
  over complete-but-fabricated state.

This is graceful degradation: CertiCoach keeps functioning wherever
possible (practice generation, spaced repetition, plan structure) while
narrowing exactly the surface — new factual claims about the exam — that
cannot be trusted without a source.
